# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workflow Rules

- **After completing any planned task:** update `CLAUDE.md` (Key Files table, Commands, etc.) and any relevant `docs/` files to reflect new files, components, or behavioral changes introduced by the task.
- **Before authoring any new homework (or editing an existing one's answers):** independently solve every problem and verify each baked-in answer is both **correct** and **complete** before committing it. Work each problem from scratch (compute numerics with a script, not by hand, to avoid arithmetic slips), inspect any referenced `figure` to confirm the values/angles you assume match what students see, and confirm `graph` key points lie on the curve the answer implies. Do not trust the source answer key тАФ instructor keys are known to contain errors and omissions. Treat a `text`/`math` answer as incomplete unless it states the full reasoning a student is expected to give. Set an explicit `sigFigs` on **every** numeric answer/part (matching the precision of the problem's given data) so the revealed correct answer shows the right significant figures тАФ a bare number literal silently drops trailing zeros (`9.00` тЖТ "9"). **Whenever your verified value differs from the instructor's source key тАФ even within the ┬▒2% tolerance тАФ log it in [docs/answer-key-discrepancies.md](docs/answer-key-discrepancies.md)** (flagging whether the gap exceeds ┬▒2%) so the instructor can fix the source key later. See [docs/homework-roadmap.md](docs/homework-roadmap.md) ┬з Authoring.
- **When authoring a homework set, explicitly analyze every problem/part for whether a standard deterministic numeric answer actually serves it well, and call out the exceptions** (in your summary to the user AND in the roadmap's per-assignment note). Watch for: (1) **ill-conditioned numerics** тАФ answers hypersensitive to rounding or where the "correct" value is тЙИ0 (no meaningful ┬▒2% band); grade these as a conceptual `text` part instead. (2) **questions whose real content is a diagram, sketch, or direction**, not a number тАФ use `answerType: "graph"` (curves) or `answerType: "vector"` (arrows / free-body diagrams), or split a "magnitude and direction" question into a numeric magnitude + a `text`/numeric direction. (3) **answers that are expressions or multi-step reasoning** тАФ use `math` (LaTeX) or `text`. Do not force a question into a numeric blank when one of these fits better; flag any part that no existing `answerType` serves so the user can decide. See [docs/homework-roadmap.md](docs/homework-roadmap.md) ┬з Authoring.

## Key Patterns

### Theme тАФ `useTheme()` in every component
All components must call `useTheme()` тАФ never import the static `s`, `BG`, `CARD`, etc. constants directly in components (those are App.jsx-only legacy usage):
```js
const { s, text, muted, border, teal, card, bg, isLight } = useTheme();
```
Every screen branch in `App.jsx` is wrapped in `<ThemeContext.Provider value={appTh}>` so all descendants pick up the correct light/dark state.

**Modal inner cards** must override `card`'s semi-transparent light-mode value with:
```js
const solidBg = isLight ? "#fff" : "#252627";
// <div style={{ ...s.card, background: solidBg, ... }}>
```

### Navigation тАФ `screen` state, no router
The entire app is one component (`App.jsx`). Navigation is `setScreen("...")` state. There are no URL routes. Within portals, `studentSection` / `instructorSection` selects the active panel. Selectors in tests must use visible text, not URLs.

### Adding new Firebase per-class state
Every new Firebase node under `classes/{classId}/` requires changes in **three places** in `App.jsx`:
1. **Startup cache restore** (~line 300): `if (c.newThing && ...) setNewThing(c.newThing);`
2. **`loadClassData` Promise.all** (~line 344): add `fbGet(classPath(classId, 'newThing')).catch(() => null),` and destructure + normalize the result
3. **`setClasses` cache call** (~line 436): add `newThing: newThingObj` to keep the in-memory class cache in sync for fast class-switching

Also add the new node name to `database.rules.json` under `$classId`.

**Exception тАФ `hwDrafts` / `hwAttempts`:** student homework drafts (`classes/{classId}/hwDrafts/{studentId}/{hwId}`) and authoritative per-item attempt counts (`classes/{classId}/hwAttempts/{studentId}/{hwId}`) are written and read directly by `HomeworkRunner.jsx` on demand (not loaded into the App.jsx class cache), since they are per-student. `hwAttempts` is the anti-gaming source of truth: written on every submit, seeded into local state unconditionally on mount, and cleared only on final submission (never by leaving/resuming).

**Homework written-work integrity check:** before final submit, students must upload images/PDFs of their handwritten work (proof of original effort). These upload to Storage at `classes/{classId}/hwWork/{studentId}/{hwId}/{file}` and are recorded on the submission as `workFiles[]`; `checkWorkIntegrity` (homework.js) runs a lenient Claude sniff-check and writes `submission.integrity = { flagged, reason, checkedAt, model, error? }`. **A flag never withholds credit on its own:** a flagged submission counts at **full credit** (and the student sees their normal score) until the instructor opens it in the Gradebook and explicitly **upholds** the flag (50% penalty); the instructor may also **clear** it (an explicit full-credit record). The decision is stored at `gradeOverrides[studentId][hwId].integrityReview = "cleared"|"upheld"`. Penalty logic flows through the shared `integrityState` (returns `{ flagged, review, penalized }` тАФ penalized only when upheld) / `integrityAdjustedScore` helpers (homework.js), used by both `Gradebook.jsx` and `StudentGrades.jsx`. The Gradebook still shows a red `*` marker on flagged cells so instructors know to review; students never see the AI verdict (the shared `SubViewModal` is passed `showIntegrity={false}` in the student view). No new RTDB node тАФ it rides on `submissions` + `gradeOverrides`; only `storage.rules` gains the `hwWork` path.
